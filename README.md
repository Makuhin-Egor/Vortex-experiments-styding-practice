## 🎯 Цель исследования

Освоение стека симуляции проекта Vortex, проведение параметрического анализа конфигураций и оценка эффективности масштабирования вычислительных ядер и кэш-иерархии с точки зрения **производительности** и **потребления ресурсов ПЛИС**.

Результаты необходимы для понимания архитектурных особенностей Vortex при выполнении задач, интенсивно работающих с памятью, и позволяют избежать неэффективного использования ресурсов ПЛИС (LUT, BRAM) при конфигурации GPU для задач с поточным характером доступа к памяти.

---

## 🛠 Методология

### Среда и конфигурация
- **Симулятор:** `simx` из состава Vortex (не является строго потактово точным *cycle-accurate*, но обеспечивает детерминированность).
- **Оценка ресурсов ПЛИС:** Аналитическая оценка для Xilinx (`1 BRAM = 4 КБ`).
- **Базовая конфигурация ядра:** 4 варпа (*warp*), стандартный конвейер Vortex.

### Бенчмарки

| Бенчмарк | Характеристика нагрузки | Локальность данных |
| :--- | :--- | :--- |
| `vecadd` | Чувствительна к пропускной способности памяти | Идеальная пространственная, нулевая временная |
| `sgemm` | Высокая арифметическая интенсивность | Высокая временная локальность (переиспользование блоков) |

### Оси экспериментов
1. **Strong Scaling** — фиксированный объём данных, рост числа ядер (сравнение паттернов доступа).
2. **Cache Scaling** — варьирование L1, включение L2 (1 МБ) и L3 (2 МБ) кэшей.
3. **Weak Scaling** — пропорциональное увеличение данных и ядер (проверка конкуренции за подсистему памяти).

## 🚀 Как воспроизвести

> Для запуска пайплайна убедитесь, что у вас собран проект **Vortex** и установлен менеджер пакетов **uv**.

### 1. Клонирование репозиториев
Клонируйте этот репозиторий и репозиторий Vortex на один уровень:

```bash
git clone https://github.com/Makuhin-Egor/Vortex-experiments-styding-practice
git clone https://github.com/vortexgpgpu/vortex.git
```

### 2. Сборка Vortex

```bash
cd vortex/build && make
```

### 3. Настройка окружения Python
Вернитесь в папку исследования и установите зависимости с помощью `uv`:

```bash
cd ../../vortex-experiments-styding-practice
uv sync
```

### 4. Запуск симуляций
Запустите симуляции (занимает **~15–20 минут**). Скрипт автоматически пересобирает Vortex под разные конфигурации:

```bash
bash scripts/run_all_experiments.sh
```

### 5. Извлечение метрик и построение графиков
Соберите данные из логов в CSV и постройте 4 графика:

```bash
uv run python3 scripts/parse_logs.py
uv run python3 scripts/plot_graphs.py
```

## 📊 Ключевые результаты и анализ

### 1. 🧱 Ограничения масштабирования (Strong Scaling)

Увеличение числа ядер с **1 до 4** на задаче `vecadd` дало ускорение всего **1.05×** (вместо идеальных 4×). Наблюдаемый эффект может быть связан как с ограничениями пропускной способности подсистемы памяти, так и с неоптимизированным размером рабочих групп (*work-group size*) OpenCL, что препятствует объединению запросов (*memory coalescing*).

Задача `sgemm` показала ускорение **1.51×** на 4 ядрах, однако оно не достигает идеального из-за того, что объёмов L1 (16 КБ) недостаточно для покрытия рабочих множеств матриц, и ядра сталкиваются с задержками при доступе к L2.

---

### 2. 🎭 Влияние иерархии кэшей (Cache Scaling)

Включение **L2 (1 МБ)** для одного ядра `vecadd` дало **+15%** ускорения, но на **4 ядрах** привело к **деградации производительности**. **L3 (2 МБ)** не дал прироста.

> **Анализ:** `vecadd` лишён временной локальности. Потоковые данные заполняют (*pollute*) L2/L3 без выигрыша от попаданий, а на множестве ядер создают конкуренцию за арбитраж шины L2 (*Contention*) и конфликты по наборам кэша (*Cache Set Conflicts*). Для контраста, `sgemm` обладает временной локальностью, но также ограничен задержками доступа к L2.

---

### 3. 🚧 Конкуренция за подсистему памяти (Weak Scaling)

При пропорциональном росте объёма данных и ядер (16K/1 ядро → 64K/4 ядра) время выполнения росло **почти линейно** (от 622 тыс. до 2.38 млн тактов). Подсистема памяти создает конкуренцию при параллельном доступе, даже при отсутствии пересечения данных между ядрами.

---

### 4. 💾 Оценка ресурсов ПЛИС (Аппаратный ROI)

| Конфигурация | BRAM (аналит.) | Прирост скорости для streaming |
| :--- | :---: | :--- |
| 1 ядро, 16 КБ L1 | 8 | baseline |
| 4 ядра + L2 + L3 | **800** | ❌ нет |

> ⚠️ Инвестиции в кэш-иерархию не приводят к росту производительности для задач с нулевой временной локальностью без учёта паттернов доступа и оптимизации программной модели.

## 💡 Выводы

При выбранных конфигурациях наблюдалось ограничение масштабирования для задач, чувствительных к пропускной способности памяти, обусловленное подсистемой памяти и неоптимизированными параметрами запуска OpenCL-ядер. Увеличение объёма кэш-иерархии неэффективно для задач с нулевой временной локальностью.

Дальнейшее улучшение производительности возможно за счёт:

1. 🔹 **Оптимизации программной модели** (настройка *work-group size* для *memory coalescing*).
2. 🔹 **Увеличения пропускной способности** подсистемы памяти.
3. 🔹 **Использования программной предвыборки** (*Software Prefetching*).
