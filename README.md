# 🔬 Исследование архитектуры GPGPU Vortex: Масштабирование и узкие места памяти

---

## 🎯 Цель исследования

Исследовать эффективность масштабирования вычислительных ядер и иерархии кэшей в открытой **GPGPU-архитектуре Vortex** (на базе RISC-V ISA) при помощи RTL-симуляции. Оценить влияние конфигураций на:

- **Производительность** — `Cycles`, `IPC`
- **Потребление ресурсов ПЛИС** — `BRAM`

---

## 🛠 Методология

### Среда
- **Эмулятор:** `simx` (RTL-симуляция)
- **Аналитическая оценка ресурсов ПЛИС:** Xilinx (`1 BRAM = 4 КБ`)

### Бенчмарки

| Бенчмарк | Тип нагрузки | Характеристика |
| :--- | :--- | :--- |
| `vecadd` | **Memory-bound** | Потоковая задача, данные не переиспользуются |
| `sgemm` | **Compute-bound** | Умножение матриц, высокая локальность данных |

### Оси экспериментов
1. **Strong Scaling** — фиксированный объём данных, увеличение числа ядер (сравнение memory vs compute).
2. **Cache Scaling** — варьирование L1, включение L2 (1 МБ) и L3 (2 МБ) кэшей.
3. **Weak Scaling** — пропорциональное увеличение данных и ядер (проверка пропускной способности шины).

---

## 🚀 Как воспроизвести

> Для запуска пайплайна убедитесь, что у вас собран проект **Vortex** и установлен **Python 3** с библиотеками `pandas` и `matplotlib`.

### 1. Клонирование репозиториев
Клонируйте этот репозиторий и репозиторий Vortex на один уровень:

    git clone <ссылка_на_этот_репо>
    git clone https://github.com/vortexgpgpu/vortex.git

### 2. Сборка Vortex

    cd vortex/build && make

### 3. Запуск симуляций
Вернитесь в папку исследования и запустите симуляции (занимает **~15–20 минут**):

    bash scripts/run_all_experiments.sh

### 4. Сбор данных
Соберите данные из логов в единую CSV-таблицу:

    python3 scripts/parse_logs.py

### 5. Построение графиков
Постройте графики (4 графика сохранятся в `results/plots/`):

    python3 scripts/plot_graphs.py

## 📊 Ключевые результаты и анализ

### 1. 🧱 Стена памяти: Memory-bound vs Compute-bound (Strong Scaling)

При увеличении числа ядер с **1 до 4** на потоковой задаче `vecadd` ускорение составило всего **1.05×** (вместо идеальных 4×). IPC ядра упал с `0.79` до `0.21`.

Однако compute-bound задача `sgemm` на 4 ядрах показала ускорение **1.51×**.

> 💡 Это доказывает, что вычислительные блоки Vortex масштабируются корректно, а проблема `vecadd` **исключительно в узком канале передачи данных из памяти** (Memory Wall).

---

### 2. 🎭 Иллюзия больших кэшей (Cache Scaling)

- Включение **L2-кэша (1 МБ)** для одного ядра `vecadd` дало **+15%** ускорения.
- Но на **4 ядрах** привело к **деградации производительности**.
- Добавление **L3-кэша (2 МБ)** не дало вообще ничего.

**Анализ:** `vecadd` — потоковая задача (*streaming*). Данные не переиспользуются, поэтому большие объёмы кэша бесполезны. На 4 ядрах L2 становится ареной конфликтов (**Cache Thrashing**) — ядра вытесняют данные друг друга.

---

### 3. 🚧 Перегрузка шины (Weak Scaling)

При пропорциональном росте объёма данных и ядер:

| Конфигурация | Такты |
| :--- | ---: |
| 16K / 1 ядро | 622 тыс. |
| 32K / 2 ядра | ↗ |
| 64K / 4 ядра | 2.38 млн |

Время выполнения росло **почти линейно**. В идеале время должно оставаться константой. Это подтверждает, что **общая шина памяти не справляется с параллельными запросами**.

---

### 4. 💾 Оценка ресурсов ПЛИС (Аппаратный ROI)

| Конфигурация | BRAM | Прирост скорости для streaming |
| :--- | ---: | :--- |
| 1 ядро, 16 КБ L1 | 8 | baseline |
| 4 ядра + L2 + L3 | **800+** | ❌ нет |

> ⚠️ **Инвестиции в кэши не окупаются.**

## 💡 Выводы

Для архитектур **GPGPU**, ориентированных на потоковую обработку данных, масштабирование количества **SM (ядер)** и объёма кэша упирается в **жёсткий предел пропускной способности памяти**.

Дальнейшее улучшение производительности возможно только за счёт:

1. 🔹 Увеличения **ширины шины памяти**.
2. 🔹 Внедрения механизмов **Memory Coalescing** (объединения запросов от разных потоков).
3. 🔹 Использования **программной предвыборки** (*Software Prefetching*).
